A field guide to the bridge.

A subprocess bridge between SillyTavern and Claude Code CLI. SillyTavern sends OpenAI-compatible requests to this server; the bridge translates them, injects narrative-focused system prompts, pipes the result through claude -p, and returns the response. Everything in this interface controls how that translation works.

One endpoint. Six tabs. Per-character memory, editable prompts, and whatever you add next.

Getting Started

Endpoint
Point SillyTavern's OpenAI-compatible API at http://localhost:5001/v1. Any API key string works — the bridge doesn't check.
Model
SillyTavern's model selector is ignored. Choose Opus, Opus 4.6, or Sonnet in the Settings tab instead.
Start command
Run run_bridge.bat (or python claude_bridge.py) and keep the terminal visible — structured logs appear there while the bridge is running.
Changing the port
The bridge binds :5001 by default. Change it in the Settings tab — the value persists to bridge_settings.json, but you'll need to restart the bridge for the new port to take effect (the GUI shows a Restart bridge reminder when there's a pending change).

The Six Tabs

I · Welcome
This page. Documentation, orientation, reference.
II · Settings
Model selection, effort level, creativity mode (a prompt-based stand-in for temperature, since Claude Code CLI doesn't expose the real parameter), and toggles for thinking visibility, debug logging, and tool calling.
III · System Prompt
The bridge injects a "narrative writer, not coding assistant" preamble before every request. Override it here if you want different framing. Leave empty to use the default — the default is fetched from the backend, so there's only one source of truth.
IV · Tools
Auto-Summary (the important one): maintains a running narrative digest per character. Cuts per-request usage massively on long RPs — instead of re-sending 200 raw messages every turn, you send summary + last 15. One summarization call pays back over dozens of subsequent turns, and prompt caching makes the stable parts near-free. Chunking: one-shot manual reset that rebuilds a character's summary from scratch. Summary Cache: inspector for every stored summary, one row per character, with per-entry delete.
V · Lorebook
Auto-generates World Info entries in the background after each RP message, using Sonnet for efficiency. Deep Analysis scans a full chat file for a more thorough pass. Manual Entry lets you add lore by hand.
VI · Test
Send a message directly at the bridge without going through SillyTavern. Useful when debugging or checking that a new system prompt is wired up correctly.

Per-Character Summaries

Every character gets its own summary bucket. The bridge identifies a character by hashing the first assistant message (the character's greeting) and stores summaries under auto_<hash> cache keys. Switching characters in SillyTavern auto-swaps buckets — no manual cache clearing, and two characters can't cross-contaminate each other's narrative.

If you edit a character's greeting mid-chat, the hash changes and a new bucket appears. Rare, but worth knowing. Legacy pre-per-character entries (those with the bare auto key) auto-migrate to the current character on the next auto-summary update — you'll see a ⚠ flag in the Summary Cache panel while they exist.

Editing Prompts

The bridge's internal prompts — summarization, condensation, chunk processing — live as editable Markdown files in the prompts/ directory next to claude_bridge.py:

summarize_incremental
Runs each time the auto-summary threshold is hit. Takes new messages, returns a narrative chunk that gets appended to the existing summary.
condense
Fires when a summary grows past auto_summary_max_length characters. Narrative-preserving condensation targeting ~60% of the original length.
summarize_chunk
Used by both chunking paths (the live /v1/chat/completions chunking mode and the /api/summary/generate file-based mode) for individual chunk summaries.
condense_chronological
Final pass in the file-based summary generator when combined chunks exceed 50K characters. Organizes events chronologically across the whole chat.

Templates use Python {variable} placeholders. Edits apply on the next request — no server restart needed, because load_prompt() re-reads the file every call. If you need a literal brace in a prompt (rare), double it: {{ and }}.

The main RP system prompt is not in this folder — it's the DEFAULT_BRIDGE_SYSTEM_PROMPT constant in claude_bridge.py, editable via the System Prompt tab.

Tips & Gotchas

Images
SillyTavern sends base64 images. The bridge saves them to temp_images/ and injects a file path so Claude Code can Read them directly — no OCR step, no subprocess refusals.
Tool calling
Extensions like TunnelVision route through the tool-calling toggle. Disable it if you're getting strange outputs or the extension is in an odd state.
Streaming
Currently buffered. The bridge waits for Claude's full response before returning to SillyTavern, even when ST requests streaming. Known limitation — on the list.
Chunking
One-shot by design: enable it, send one message, the bridge splits the conversation, summarizes every chunk, seeds the character's auto-summary bucket, and auto-disables. Use it after importing a long chat, not for every turn. The Cancel button disarms it without sending a request.
Character key
The hash is 16 characters of MD5 over the first 2000 chars of the greeting. Stable across turns. If you use the backup zip to restore state, cache keys remain valid.
Debug logs
Enable Debug Output in Settings to see every incoming request structure, image detection passes, and prompt sizes in the bridge terminal. Essential when wiring up new SillyTavern extensions.

Auto-Summary

Cuts per-request token usage dramatically on long RPs. Without it, SillyTavern re-sends your entire message history every turn — at message 200, that's the whole conversation going up the pipe on every single reply. Auto-summary replaces old messages with a condensed narrative digest the bridge keeps per character, so each request sends summary + last 15 messages instead of the full backlog. The one-time cost of generating the summary is paid back many times over the next dozens of turns, and because the summary + system prompt stay stable across requests, Claude's prompt cache kicks in and most of those tokens come back as cheap cache reads instead of full reads. New events get appended without losing old ones, and each character's story lives in its own bucket — switching characters won't cross-contaminate context.

How It Works

  1. Click "Enable Chunking" above
  2. Go to SillyTavern and send any message
  3. The conversation is split into ~80K token chunks
  4. Each chunk is summarized (cached for reuse)
  5. Summaries are combined into context
  6. Your request is sent with the summary context
